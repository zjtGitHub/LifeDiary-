# 日记与人生表格应用项目计划书

## 1. 项目概述

项目暂定名：**时光日记**

本项目是一款面向个人长期记录的日记与人生管理应用，核心功能包括每日文字日记、图片记录、日历视图、待办计划、标签搜索、备份恢复，以及以“一周一个方块”呈现的 100 岁人生表格。

产品目标不是做一个单纯的日记本，而是帮助用户把每天的记录、计划、照片和人生进度连接起来，形成一个可长期沉淀、可回顾、可备份的个人时间系统。

## 2. 产品定位

### 目标用户

- 有长期记录习惯的个人用户。
- 希望通过日历、todo 和周回顾管理生活的人。
- 注重隐私和数据所有权的用户。
- 喜欢人生进度、年度复盘、长期目标管理的用户。

### 核心价值

- **记录**：快速记录每天的文字、图片和计划。
- **回顾**：通过日历、搜索、标签、周回顾找回过去。
- **觉察**：通过人生表格看到时间流逝和当前阶段。
- **安全**：支持本地优先、备份导出，后续扩展云同步和加密。

## 3. 功能规划

### 当前 MVP

- 每日纯文本日记。
- 每日多图上传、预览和删除。
- 月历选日。
- 按日期管理 todo。
- 标签和搜索。
- 人生表格，按 100 岁计算每周方块。
- 点击人生周跳转到对应日期并显示周回顾。
- JSON 备份导入导出。
- 移动端响应式适配与 PWA 基础能力。

### V1 商业版

- 用户账号系统。
- 多设备云同步。
- 自动云备份。
- 图片云存储。
- PIN / 生物识别锁。
- 全文搜索增强。
- 日历状态统计。
- 周报、月报、年报。

### V2 增强版

- 端到端加密。
- AI 每周总结、月度总结、年度回顾。
- 情绪趋势和生活关键词分析。
- 模板系统：晨间日记、复盘、旅行、读书、健身。
- 导出 PDF / Markdown / 年度书。
- 桌面端应用。

## 4. 技术架构

### 总体架构

商业版本建议采用：

```text
客户端 App
  -> 本地离线数据库
  -> 同步层
  -> 后端 API
  -> 云数据库 / 图片对象存储 / 备份系统
```

### 客户端策略

第一阶段建议复用当前 Web 应用：

- Web：React + Vite + TypeScript。
- 移动端：Capacitor 打包 Android / iOS。
- 桌面端：后续可用 Tauri 或 Electron。
- 本地存储：当前使用 IndexedDB；商业移动端可逐步迁移到 SQLite。

长期商业化后，可考虑：

- 移动端：React Native / Flutter。
- PC 端：React Web + Tauri / Electron。
- 共享层：日期计算、人生表格算法、同步协议、数据模型。

### 后端架构

推荐后端能力：

- API 服务：Node.js / Go / Java / Python 均可。
- 主数据库：PostgreSQL。
- 图片存储：S3 / Cloudflare R2 / 阿里云 OSS。
- 缓存和任务队列：Redis。
- 搜索：PostgreSQL 全文搜索起步，后续可接 Meilisearch / Elasticsearch。
- 鉴权：邮箱登录、OAuth、设备管理。

### 数据安全

最低要求：

- HTTPS。
- 服务端加密存储敏感数据。
- 图片私有桶。
- 定期备份。
- 用户可导出全部数据。

进阶能力：

- 端到端加密。
- 本地应用锁。
- 恢复密钥。
- 加密备份包。

## 5. 商业模式

### 推荐模式

采用 **免费本地版 + 订阅高级功能**。

免费功能：

- 本地日记。
- 图片记录。
- todo。
- 日历。
- 人生表格。
- 手动备份导出。

付费功能：

- 云同步。
- 自动备份。
- 多设备使用。
- 更多图片空间。
- 高级搜索。
- AI 总结。
- 加密空间。
- PDF / 年度报告导出。

### 定价建议

- 免费版：本地使用。
- Pro 月付：12-18 元 / 月。
- Pro 年付：98-148 元 / 年。
- 一次性本地高级版：68-128 元。
- 云存储扩容：按容量阶梯收费。

### 不建议模式

- 不建议广告变现。
- 不建议出售用户数据。
- 不建议在隐私内容旁做推荐流。

日记产品的核心资产是信任，广告和数据交易会严重损害产品定位。

## 6. 版本路线

### 阶段 1：个人可用版

目标：当前项目变成稳定可长期使用的本地应用。

交付：

- 完善移动端 App 打包。
- 优化图片存储和压缩。
- 完善备份恢复。
- 修复移动端细节。
- 增加基础密码锁。

### 阶段 2：小规模商业测试

目标：验证用户是否愿意长期使用并付费。

交付：

- 账号系统。
- 云同步。
- 自动备份。
- 简单订阅。
- 数据导入导出。
- 用户反馈入口。

### 阶段 3：正式商业版

目标：面向公开用户发布。

交付：

- Android / iOS 应用。
- Web / 桌面端。
- 稳定同步。
- 图片云存储。
- 付费订阅。
- 隐私政策和用户协议。
- 应用商店上架材料。

### 阶段 4：增长与高级功能

目标：提高留存和付费转化。

交付：

- AI 周报和年报。
- 模板市场。
- 年度书导出。
- 高级统计。
- 家庭 / 情侣 / 私密共享空间。

## 8. 风险与应对

### 数据丢失风险

应对：

- 本地自动备份。
- 云端版本历史。
- 用户主动导出。
- 恢复演练。

### 隐私信任风险

应对：

- 明确隐私政策。
- 不接广告。
- 不出售数据。
- 支持数据导出和删除。
- 后续加入端到端加密。

### 多端同步复杂度

应对：

- 先做单用户增量同步。
- 每条数据带 `updatedAt` 和版本号。
- 冲突时保留多个版本。
- 图片和文本分开同步。

### 获客困难

应对：

- 先做小众精准定位。
- 强调隐私、本地优先、人生表格。
- 做年度总结、人生格子图等可传播内容。
- 通过小红书、B 站、独立开发者社区验证需求。

## 9. 团队配置

最小团队：

- 1 名前端 / 全栈工程师。
- 1 名后端工程师。
- 1 名产品设计。

独立开发者模式：

- 先用当前 React 项目 + Capacitor + 简单后端。
- 优先做 Android。
- iOS 在产品验证后再投入。

## 10. 近期执行建议

建议下一步按这个顺序做：

1. 使用 Capacitor 打包 Android App。
2. 增加图片压缩，降低本地和云端存储压力。
3. 设计云同步数据模型。
4. 增加账号和自动备份。
5. 做一个可公开测试的 Beta 版本。
6. 收集 20-50 个真实用户反馈。
7. 决定是否投入 React Native / Flutter 原生重写。

## 11. 项目判断

这个项目适合从“本地隐私工具”切入，而不是一开始做大而全的效率软件。

最有商业潜力的卖点是：

- 私密可信。
- 长期记录。
- 多设备安全保存。
- 人生表格和周回顾带来的独特情感价值。
- AI 总结带来的后期增值。

推荐商业路径：

```text
免费本地版
  -> 移动 App
  -> 云同步订阅
  -> AI 总结和年度报告
  -> 隐私加密高级版
```
